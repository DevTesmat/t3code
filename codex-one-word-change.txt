The project begins with a practical idea and shapes it into a dependable working instrument.
Every screen should respond promptly, even when the system is under sustained pressure.
Every message should arrive in order, even when a provider reconnects in the middle of a stream.
Every session should feel predictable, even after a restart, resume, or interrupted turn.

The server coordinates provider instances, workspace access, terminal streams, checkpoints, and event delivery.
The web app turns those events into clear state for the person using the product under real workload.
The desktop shell provides the native edges: folders, environment discovery, update flow, and packaging.
The shared contracts keep each boundary explicit when data moves between packages or across the wire.

Reliability is a primary concern in this codebase, not a polish item.
It is the central condition for trust in a tool that brokers long-running agent work.
When a process fails, the application should describe what happened, preserve useful context, and keep recovery visible.
When a connection drops, the application should recover without confusing the current session or hiding partial progress.
When history is loaded, the application should avoid duplicate events, missing events, stale status, and misleading ordering.

Performance matters because slow instruments interrupt thought and make feedback loops expensive.
Maintenance matters because early shortcuts become future constraints that slow every later feature.
Small changes should still respect the shape of the system, the responsibilities of each package, and the cost of future debugging.
Clear contracts should make the next change easier to review, test, reason about, and operate under failure.
